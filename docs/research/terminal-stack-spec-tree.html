M.Map

Implementation Map — Rust Crate Interconnections

How the crates from the spec tree actually wire together when you build a terminal in Rust. Every arrow is a real "depends on" or "calls into" relationship. Solid = uses directly. Dashed = wraps / composes. The ferris-orange line is rustix skipping libc entirely.

Vertical depth between any two nodes is roughly how many crate boundaries a single byte crosses. Hub crates (ferris-orange borders) are the spine: nix, vte, termwiz, tokio, mio, portable-pty. The two ferris-orange paths from rustix and tokio-uring straight to the kernel show the only routes in this graph that bypass libc entirely.

Five patterns worth pulling out of the map:

The libc spine
Most paths converge here
L04 → L01 → L00
Almost every arrow eventually funnels through nix → libc → kernel. The exceptions are rustix and tokio-uring, which issue raw Linux syscalls. On macOS and BSD, even those typically need libc because syscall ABIs aren't stable there.
vte is the parsing hub
Most L08–L10 crates feed it
L06 central
OSC handlers (arboard, linkify, notify-rust), input parsing (termwiz::input), and graphics (viuer) all consume vte's parsed events. termwiz is the only alternative — it has its own parser internally.
portable-pty is the cross-platform bridge
L02 composition node
L02
It wraps nix::pty on Unix and windows::Console on Windows behind one trait-based API. Without it, you'd be writing #[cfg(unix)] / #[cfg(windows)] at every PTY operation.
tokio dominates async
All L13 protocols ride it
L04 async hub
tower-lsp, rmcp, jsonrpsee, jupyter-protocol, dap-types — every L13 protocol crate runs on tokio. tokio uses mio, which abstracts epoll / kqueue / IOCP by calling libc (or windows-sys) directly. Note that mio and nix are siblings — both wrap libc, neither sits on top of the other. One async runtime, three OS event mechanisms beneath it, two parallel safe-Rust paths into libc.
The non-Rust dashed line
External shells aren't replaceable
L11 boundary
The orange dashed arrow from std::process::Command to "external: bash · zsh · fish · pwsh" is where the Rust-only assumption breaks. You can replace almost everything else with Rust, but if your users want bash compatibility, you're shelling out to a C binary. Nushell sidesteps this — but only by giving up POSIX shell semantics.

Minimum Viable Modern Terminal

If you're building a "modern Linux-like multiplatform terminal," this is the minimum implementation surface to cover ~95% of what modern programs expect — plus room for a novelty layer on top.

// IMPLEMENTATION CHECKLIST
OS interface
POSIX.1 + ConPTY (portable-pty if Rust)
PTY
Unix98 (posix_openpt) + ConPTY
Line discipline
termios (POSIX) — full c_iflag/c_oflag/c_lflag
Encoding
UTF-8 + UAX #11 + UAX #29
Escape codes
ECMA-48 + xterm ctlseqs superset
OSC minimum
0/1/2 · 4 · 7 · 8 · 52 · 133 (10/11/12, 633, 1337 for parity)
Graphics
Sixel and/or Kitty graphics protocol
Input
SGR mouse 1006 · bracketed paste 2004 · Kitty keyboard
Identification
DA1 · XTVERSION · terminfo entry
Truecolor
CSI 38;2;r;g;b m + advertise $COLORTERM=truecolor
Shell integration
OSC 133 emitter scripts: bash / zsh / fish / pwsh
Structured channel
Pick a private OSC code, define a JSON/MessagePack/CBOR schema, ship graceful fallback to plain text
